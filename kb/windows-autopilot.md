# Windows Autopilot

**Summary**: Windows Autopilot is a cloud-based service that automates deployment and configuration of new Windows devices with minimal IT intervention; this page covers what Autopilot actually does at the protocol level, including OOBE, Entra ID join, MDM enrollment, and IME installation.

**Sources**: `Onboarding modern with Autopilot Magic trick revealed.md`

**Last updated**: 2026-04-13

---

## What Autopilot Is (and Isn't)

Autopilot is a cloud service that automates Windows 10/11 device deployment and configuration for a ready-to-use state with minimal IT intervention. It is *not* an imaging tool — it relies on the OEM Windows installation and applies configuration through MDM.

Autopilot's job ends once the JSON profile is downloaded and the device reaches the login prompt. Everything after that is Windows, OMA-DM, and the [[intune-management-extension]]. (source: Onboarding modern with Autopilot Magic trick revealed.md)

## Prerequisites

- Device hash uploaded to the Autopilot database (by admin or vendor)
- Group tags applied if using dynamic Entra ID group membership
- Dynamic group query examples:
  - All Autopilot devices: `(device.devicePhysicalIDs -any _ -contains "[ZTDId]")`
  - Specific group tag: `(device.devicePhysicalIDs -any _ -eq "[OrderID]:EntraID")`
- Upload the hash at least **one day before** provisioning to allow dynamic group membership to sync

(source: Onboarding modern with Autopilot Magic trick revealed.md)

## Phase 0: OOBE and Autopilot Profile Retrieval

When the device first connects to the internet during OOBE:

1. Device contacts `login.live.com` with a device-add request using a randomly generated username/password generated by `wlidsvc.dll` (Microsoft Account Sign-In Assistant)
2. `login.live.com` returns a hardware device ID
3. An `rsts2.srf` token request goes to `login.live.com` → returns a final device token
4. Device token is stored locally at `HKCU:\SOFTWARE\Microsoft\IdentityCRL\Immersive\production\Token`
5. Device contacts `ztd.dds.microsoft.com` (Autopilot service) with the device token to request the Autopilot profile
6. If the device is in the Autopilot database, the profile (JSON) is downloaded and saved to `C:\Windows\servicestate\wmansvc`
7. Windows Management Service reads the JSON and displays company branding; user sees the login screen

> Note: `ztd.dds.microsoft.com` is not listed in the standard Intune endpoints doc — it is in the Autopilot-specific requirements doc.

(source: Onboarding modern with Autopilot Magic trick revealed.md)

## Phase 1: Entra ID Join

After the user enters credentials:

1. `Microsoft.Windows.CloudExperienceHost_cw5n1h2txyewy` (web app) connects to `login.microsoftonline.com/common/oauth2/token` to request an Entra token; also requests the MDM token if the user has an Intune license
2. `dsreg.dll` locates the Entra ID join URL and contacts `enterprise.registration.net/yourdomain/discover?api-version=1.7`
3. Response includes the `DeviceJoinService` URL (`enterpriseregistration.windows.net/EnrollmentServer/device/`) and `JoinResourceID`
4. Device sends a join request with the token to the Device Join Service
5. Device Join Service validates the ID token, creates the device object in Entra ID, and responds with the **ms-organization certificate**
6. Response also includes user SID and group SIDs to add to the local Administrators group

(source: Onboarding modern with Autopilot Magic trick revealed.md)

## Phase 2: MDM Enrollment (Intune)

1. Username entered during Entra ID join resolves the MDM discovery URI (`enrollment.manage.microsoft.com`)
2. Device contacts the discovery service with the domain email → receives the actual enrollment URI
3. Device obtains a security token containing DeviceID, TenantID, IP Address, and ZTD device ID
4. Device contacts the MDM service with the token → receives a blueprint for an x509 certificate
5. Device requests the certificate using the security token → receives certificate + designated storage location
6. Certificate is stored on the device **and in the TPM** → enrollment complete; device and Intune share trust

(source: Onboarding modern with Autopilot Magic trick revealed.md)

## Phase 3: Preparing for MDM

The `OmaDMClient.exe` process:

1. Queries NodeCache version at `HKLM\SOFTWARE\Microsoft\Provisioning\NodeCache\CSP\Device\MS DM Server\CacheVersion`
2. Enumerates available CSPs from four registry paths under `HKLM\SOFTWARE\Microsoft\Provisioning\CSPs`
3. Enumerates policy providers and security settings
4. Changes the **Device Management Wireless Application Protocol** service from Manual to Automatic Delayed Start
5. Writes `RequireRetrieveHealthCertificateOnBoot` to both the providers and current policy hives (for health attestation)
6. TPM service contacts `has.spserv.microsoft.com` to obtain a health certificate
7. NodeCache is populated (~1308 settings on a typical device; ~2.5 minutes to complete)

### IME Installation

`EnterpriseDesktopAppManagement` CSP delivers the IME:
- Registry path: `HKLM\SOFTWARE\Microsoft\EnterpriseDesktopAppManagement\S-0-0-...\MSI\%GUID%`
- `MDMAppInstaller.exe` installs the IME from MSI
- ESP tracks installation via a tracked registry path
- The same CSP is used for Endpoint Privilege Management (EPM) client and CM co-management setup

### ADMX Policies

1. `OmaDMClient.exe` downloads ADMX templates and ingests them one by one
2. Ingested policies are written to `HKLM\SOFTWARE\Microsoft\PolicyManager\AdmxDefault\%Intune MDM GUID%\`
3. Completion is recorded with a timestamp in the same registry hive

(source: Onboarding modern with Autopilot Magic trick revealed.md)

## Phase 4: Device Setup — Policy Delivery Order

Security policies, certificates, and network configurations are applied. The observed delivery order (may vary by tenant configuration):

| Order | Policy | ESP Tracked? | Notes |
|-------|--------|-------------|-------|
| 1st | Certificates | Yes | REG_BINARY `Blob` key written to SystemCertificates |
| 2nd | DeviceHealthMonitoring | No | Written to providers and current hives |
| 3rd | Defender (signature/engine/platform rings) | No | Applied even without explicit configuration |
| 4th | Windows Updates | No | `AllowAutoUpdate` and update ring policies |

(source: Onboarding modern with Autopilot Magic trick revealed.md)

## Troubleshooting Tips

- Company branding not showing → device hash not found in Autopilot database, or dynamic group not resolved yet
- Script `[PowerShell] Request PS Scripts for user logon` in IME logs = session change fired successfully — see [[intune-management-extension]]
- IME install error `0x800705B4` — documented community issue with IME MSI delivery
- Use [SyncMLViewer](https://github.com/okieselbach/SyncMLViewer) to trace OMA-DM protocol stream during enrollment
- Autopilot profile is saved as JSON at `C:\Windows\servicestate\wmansvc`
